# os-smart-agent - 自然语言驱动的 Linux 运维代理

## 概述

`os-smart-agent` 是一个自然语言驱动的 Linux 运维代理，可将用户以自然语言描述的操作意图解析为结构化的系统命令，在远程 Linux 环境中安全、可控地执行并返回格式化结果。

项目遵循模块化、可配置、安全优先的设计原则，支持本地与 SSH 远程两种执行模式，当前主推"Windows 宿主机 + SSH 远程 Linux 虚拟机"的部署拓扑。

在运行项目前，你应该先在config.yaml文件中配置自己的api和ssh用户信息。

## 2. 整体架构设计

### 2.1 设计目标与原则

**目标与实现方式：**
- 意图理解灵活性：基于大语言模型（DeepSeek-V3.2）进行自然语言解析，避免硬编码规则
- 执行环境真实性：通过 SSH 将命令发送至真实 Linux 环境执行，确保结果与生产一致
- 安全可控：内置风险操作分级、二次确认机制，防止误操作引发事故
- 配置与逻辑分离：所有运行参数集中在 config.yaml，程序逻辑不变
- 分层可测试：各模块独立封装，支持单元测试与依赖覆盖验证

### 2.2 系统分层架构

系统采用分层架构，自顶向下依次为交互层、智能核心层、执行与安全层、基础设施层。

**关键特征：**
- 调度中心 `main.py` 根据配置动态选择 `LocalExecutor` 或 `SSHExecutor`
- 安全模块位于执行模块之前，所有命令必须通过安全校验才能执行

### 2.3 核心模块职责

| 模块 | 路径 | 职责 |
|------|------|------|
| 配置管理 | src/config.py | 从 config.yaml 加载并校验全部运行参数 |
| 意图识别 | src/intent/ | 调用 LLM 将自然语言转为结构化 Intent |
| 命令执行 | src/execution/ | 本地或 SSH 远程执行系统命令，返回统一结果 |
| 安全控制 | src/security/ | 识别高风险操作，触发二次确认 |
| 对话管理 | src/conversation/ | 维护多轮对话历史与上下文 |
| 结果反馈 | src/feedback/ | 格式化命令输出为可读文本 |
| 调度中心 | src/main.py | 组件初始化、交互循环、异常处理 |

### 2.4 部署拓扑

智能部分在 Windows，执行对象在 Linux，二者通过 SSH 解耦。

## 3. 各项能力实现进度与深度说明

### 3.1 配置管理

**进度：已完整实现**

**深度：**
- 基于 `pydantic` 的配置模型校验，必填项缺失即报错
- 支持本地/SSH 模式切换、目标主机、模型参数、安全策略的全配置化
- 提供 `example_config.yaml` 示例文件
- 敏感字段可写于配置文件，也预留环境变量覆盖逻辑

**局限：** 当前凭据以明文为主，环境变量覆盖非强制，后续需升级安全策略

### 3.2 意图识别

**进度：已完整实现**

**深度：**
- 调用 DeepSeek-V3.2 模型，返回结构化意图（类型、风险等级、参数、解释）
- 覆盖磁盘查询、文件查找、进程/端口查询、用户增删等常见运维场景
- 系统提示词已封装，返回格式约束为 JSON，降低解析失败率

**局限：** 仅支持单意图；意图类型集为预定义枚举，扩展需同步修改映射表

### 3.3 命令执行

**进度：已完整实现（SSH 为主，本地保留）**

**深度：**
- 统一 `execute(command, timeout)` 接口，屏蔽执行差异
- SSH 执行器基于 Paramiko，支持密码/密钥认证、连接复用、超时控制
- 返回标准化结果对象（success, stdout, stderr, command）

**局限：** 同步阻塞模式，无流式输出；未内建 sudo 管理，权限依赖登录用户

### 3.4 安全控制

**进度：已完整实现**

**深度：**
- 高/中两级风险操作清单，支持 `config.yaml` 自定义
- 高风险操作强制二次确认，中风险可按需配置确认
- 安全模块在命令执行前拦截，独立于意图识别，不依赖模型判定的准确性

**局限：** 静态清单匹配，无法动态评估未知危险；暂无审计日志持久化

### 3.5 对话管理

**进度：已实现基础功能**

**深度：**
- 维护多轮对话历史，支持上下文引用（如"再查一下 /data"）
- 对话记录包含用户输入、意图、执行结果，供下一次解析参考

**局限：** 仅在内存保留，重启丢失，不支持持久化或会话恢复

### 3.6 结果反馈与格式化

**进度：已实现基础功能**

**深度：**
- 对命令输出进行结构化封装，提取关键信息并以可读文本返回
- 支持长输出截断，避免 CLI 刷屏

**局限：** 格式化规则简单，未针对不同命令类型定制差异化模板

### 3.7 依赖管理与环境验证

**进度：已完整实现**

**深度：**
- `requirements.txt` 声明全部核心依赖

**局限：** 依赖版本范围较宽，未来可锁定兼容版本

### 3.8 多 IDE 支持与调试

**深度：**
- 已在 VS Code、PyCharm、命令行等环境验证启动，提供 `launch.json` 示例

## 4. 行为设计逻辑

### 4.1 单次交互流程

代理每次交互遵循严格的线性处理链，保证行为可控、可追踪。

**关键设计：**
- 意图识别与命令生成分离，防止模型直接输出危险命令
- 安全校验在命令执行前统一拦截，形成安全栅栏
- 每一步状态记录于对话管理，便于调试与上下文引用

### 4.2 安全确认机制

采用"两级风险 + 显式确认"模式：

- **高风险：** 删除系统目录、格式化磁盘、修改关键配置等 → 必须二次确认
- **中风险：** 删除普通文件、修改一般配置、用户权限变更 → 可配置是否确认
- **低风险：** 查询类操作 → 直接执行

**确认流程：**
系统暂停执行，展示拟执行操作及风险；
用户需明确输入 "yes"/"no"；
超时或否定则取消操作。

该设计确保执行破坏性操作前存在"人类最终裁定环节"。

系统提示词定义了代理身份、可处理意图的枚举、输出格式与风险判定标准：

```
你是 Linux 运维助手的意图解析器。
将用户输入分类为以下意图之一：
- disk_usage_query: 查询磁盘使用情况
- file_search: 查找文件
- process_query: 查询进程
- port_query: 查询端口占用
- user_create / user_delete: 用户管理
- system_info: 系统信息
- ...

必须返回 JSON 对象，包含：
{
  "intent": "意图类型",
  "risk_level": "high/medium/low",
  "parameters": { ... },
  "explanation": "简短说明"
}

风险标准：high → 删除系统文件/目录、格式化等；medium → 删除普通文件、权限变更等；low → 只读查询。
仅返回 JSON，无其他文字。
```

### 5.3 参数抽取

LLM 输出的 `parameters` 为动态键值对，例如：
输入"查找 /var/log 下所有 .log 文件" → `{ "path": "/var/log", "pattern": "*.log" }`。

参数在后续命令映射阶段填入模板，如 `find {path} -name "{pattern}" -type f`。

### 5.4 错误处理

- JSON 解析失败 → 提示"无法理解您的意图，请重新描述"
- 意图类型不在预定义集合 → 映射为 `unknown`，请求用户澄清
- API 超时或错误 → 友好报错，不强行降级

## 6. 设计取舍

下表汇总了架构、算法与实现中的关键取舍及理由。

| 设计点 | 采纳方案 | 未采纳方案 | 取舍理由 |
|--------|----------|----------|----------|
| 意图解析方法 | 基于 LLM 的提示词分类 | 纯规则匹配 / 小模型微调 | LLM 灵活性高，无需大量标注；规则过于僵化，微调成本高 |
| 命令生成方式 | 意图→命令映射模板 | LLM 直接生成命令 | 模板化确保命令经审核、可预期；直接生成存在注入风险 |
| 多意图支持 | 仅支持单意图 | 复合意图与任务流 | 当前运维场景以单任务为主，简化复杂度，提高解析可靠率 |
| 风险识别 | 静态清单匹配 | 动态风险评估 | 清单透明可解释，足够覆盖主要高危操作；动态评估不确定性更高 |
| 凭据管理 | 配置文件 + 可选环境变量 | 强制环境变量 / 密钥服务 | 开发阶段配置直接更便捷，环境变量注入逻辑已预留，可平滑升级 |
| 执行返回 | 同步阻塞，完整输出 | 流式输出 | 运维命令执行快，完整输出易于一次格式化；流式增加无谓客户端复杂度 |
| 对话持久化 | 内存保留，重启丢失 | 数据库持久化 | 当前为短时运维窗口，持久化非刚需，且降低部署依赖 |
| 前端形式 | CLI | Web 界面 | CLI 契合运维场景，开发成本低，聚焦核心逻辑验证 |
| 跨平台部署 | Windows 运行 agent + SSH 到 Linux | 全 Linux 环境 | 保留 Windows 开发便利性；Linux 作为执行环境不可替代，SSH 实现解耦 |
